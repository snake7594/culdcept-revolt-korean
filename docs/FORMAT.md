# CULDCEPT.DAT 포맷 (컬드셉트 리볼트, 3DS)

아래는 전부 게임에서 직접 리버스 엔지니어링한 내용입니다(공식 문서 없음).
오프셋·핸들러 주소는 압축 해제된 ARM 실행 파일(ExeFS `.code`, 디스크에서는 BLZ 압축됨)
기준입니다.

## 1. 아카이브 컨테이너

```
오프셋 0:  레코드 테이블 -- 8바이트 엔트리 N개:
             u32 LE  offset      (파일 내 위치)
             u32 LE  size        (이 엔트리의 바이트 수)
           N = table[0].offset / 8   (데이터는 테이블 바로 뒤에서 시작)
엔트리 i:  DAT[offset : offset+size]
```

엔트리 첫 바이트가 코덱 타입입니다. 관측된 타입: `0x08`, `0x0c`, `0x0d`, `0x8d`(압축);
`0x00`(원시/중첩 컨테이너, 24비트 빅엔디언 오프셋 테이블 사용); `0x05`(원시 CWAV 오디오).

엔트리 크기를 바꾸려면 새 바이트를 **파일 끝에 추가**하고 해당 8바이트 레코드만 갱신하면
됩니다 — 다른 엔트리의 오프셋은 그대로 유지됩니다.

## 2. 코덱 디스패치

단일 디스패처(guest `0x27f3f4`)가 타입 바이트를 읽어 핸들러 테이블(`0x39bb58`,
`type & 0x3f`로 인덱싱) + 후처리기(`0x39bb90`, `type >> 6`로 인덱싱)를 선택합니다:

| 타입 | 핸들러 | 방식 |
|------|--------|------|
| 0x08 | 0x274b90 | canonical Huffman + LZ (윈도우 0x0d) |
| 0x0c | 0x274b88 | canonical Huffman + LZ (윈도우 0x10) |
| 0x0d | 0x275080 | LZMA 계열 레인지 코더 |
| 0x8d | 0x275080 | 동일 레인지 코더, 후처리기만 다름 |

## 3. 타입 0x08 / 0x0c — Huffman + LZ (`culdcept/huffman.py`에 구현)

```
entry = [type u8] [varint size] [비트스트림]
```

- **size**는 연속 비트(continuation-bit) **varint**(핸들러 `0x2747f8`)이며, `size>>8`이
  **아닙니다**. 아카이브 레코드의 size 필드와 이 varint는 겹치는 바이트에서 읽히지만,
  실제 압축 해제 길이는 varint입니다.
- **비트 리더**(`0x275020/0x275028`): 32비트 누산기에 대해 MSB 우선. 리필은 little-endian
  `u16`을 읽어 바이트를 뒤집어 누산기 상위에 넣습니다. varint 뒤 `pos`가 홀수면 1바이트를
  미리 로드하고, `consume(16)`으로 프라이밍. 결과적으로 `pos`부터의 엔트리 바이트에 대한
  단순 MSB 우선 비트스트림입니다.
- **블록**을 `size` 바이트가 나올 때까지 반복:
  - `r8 = getbits(16)` — 블록 내 심볼 수.
  - **litlen** 테이블(`0x274d18`): 19심볼 코드길이(CL) 테이블을
    `read_lengths_direct(19, 5, special_pos=3)`으로 읽고, `count = getbits(9)`개의 코드
    길이를 CL로 RLE 디코딩(`s>2 → 길이 s-2`, `s==0 → 0`, `s==1 → 3+getbits(4)`개의 0,
    `s==2 → 0x14+getbits(9)`개의 0).
  - **dist** 테이블: `read_lengths_direct(window+1, (window+1-10)&5, -1)`.
  - `r8`개 심볼: `sym<0x100`이면 리터럴; 아니면 `length=sym-0xFD`, 거리 코드
    `dsym → dd = (dsym==0?1:(1<<(dsym-1))+1+getbits(dsym-1))`, 매치 복사.
- **`read_lengths_direct(nsym,nbits,special)`**: `count=getbits(nbits)`; 0이면 단일 심볼
  `= getbits(nbits)`. 아니면 길이마다: 상위 3비트 `t`; `t<7 → 길이 t`(3비트 소비) 아니면
  `길이 = 7 + clz32(~(acc<<3))`(길이-3 소비). `i==special`에서 `r=getbits(2)`; `r>0`이면
  `r`개의 0을 건너뜀.

최소 유효 인코더는 블록마다 `r8`, degenerate CL 테이블(단일 심볼 10 → 0비트), `count=256`
(256개 전부 길이 8 → 리터럴 바이트 B가 그대로 8비트 값 B), degenerate dist 테이블, 그리고
원시 리터럴 바이트를 씁니다. `compress()` 참고.

## 4. 비트맵 폰트 (압축 해제된 리소스, `culdcept/font.py`)

```
오프셋 0x30: CMAP -- u16 LE 문자 코드; glyph_index i -> CMAP[i].
             ASCII 0x20-0x7e, 반각 0xa1-0xdf, Shift-JIS 0x8140+, 가이지 0xf9xx.
크기 섹션 헤더(8바이트): 14 00 | bpg(u16) | b4 | w | h | bpp
```

4bpp 텍스트 크기(`bpp==4`, `b4==0xCE`)의 경우 글리프 비트맵은 `섹션 + 0xCE`에서 시작하며,
글리프 `i`는 고정 `w×h` **A4**(4비트 알파, 상위 니블 우선) 셀로 `bpg = ceil(w/2)*h` 바이트,
위치는 `섹션 + 0xCE + i*bpg`입니다.
탑재 크기: **10px**(bpg 50), **12px**(bpg 72), **14px**(bpg 98).
12×12 16bpp "아웃라인" 섹션(`b4==0x10`, `bpg 288`)도 있지만 글리프가 ~160개뿐(한자 없음)
이라 본문 텍스트에는 쓰이지 않습니다.

게임은 Shift-JIS를 CMAP으로 글리프 인덱스로 변환한 뒤 글리프를 A4 텍스처에 그려서 텍스트를
렌더링합니다. 렌더링 파이프라인은 Azahar LayeredFS로 실기 확인했습니다.

## 5. 시나리오 컨테이너 (스토리 대사, `culdcept/scen.py`)

스토리 대사는 코덱으로 압축된 최상위 엔트리에 **없습니다**. 대사는 자체 헤더로 시작하는
**시나리오 컨테이너** 엔트리에 있습니다. 이 엔트리의 첫 바이트는 코덱 타입이 아니라 헤더
크기의 최하위 바이트라서(관측값 `0x30`/`0x48`/`0x68`/`0x80`/`0xa8`/`0xb0`/`0xe0` 등), 코덱
타입만 보는 스캐너는 이 엔트리를 통째로 지나치게 됩니다.

```
dword0 (u32 LE) = 헤더 크기(바이트).  헤더 = (섹션 offset u32, 섹션 length u32) 쌍 배열.
                  섹션 개수 = dword0 / 8.
섹션 k          = entry[offset : offset+length].  대부분 0x08(간혹 0x0c) 압축.
```

한 스테이지의 컨테이너에는 여러 섹션이 있고, 그중 하나가 **대사 스크립트**입니다
(압축 해제 후):

```
[스크립트 바이트코드 / 헤더]  그 다음  [텍스트 영역]
```

- **텍스트 영역**은 null(0x00) 종료 "이벤트"의 연속이고, 각 이벤트는 `0x07`로 나뉜
  "페이지"(대화창 하나)들로 이루어집니다.
- **제어 코드**: `0x00`=이벤트 끝, `0x07`=페이지 넘김(클릭 대기), `0x0a`=줄바꿈,
  `0x03 0x30 0x2f`=플레이어 이름 삽입.
- 이벤트는 내부에 `0x00`을 포함하지 않으므로, 섹션을 null로 분할했을 때 **마지막 N개
  세그먼트가 곧 N개 이벤트**입니다 — 이렇게 텍스트 시작을 확실히 찾습니다.
- ★**게임은 각 페이지/이벤트의 시작을 텍스트 영역 내 절대 바이트 오프셋으로 참조합니다**
  (실기 확인). 따라서 텍스트 길이를 바꾸면 뒤 오프셋이 전부 어긋나 대사가 밀리거나
  뒤섞이고 화자도 어긋납니다. **모든 `0x07`/`0x00`의 바이트 위치를 원본과 동일하게
  유지**해야 합니다.

편집 절차: 대사 섹션을 압축 해제 → 스크립트 헤더는 그대로 두고 **각 페이지를 원본 페이지의
바이트 길이에 맞춰**(한국어가 짧으면 뒤를 공백 `0x20`으로 패딩; 화면엔 안 보임) 교체하여
텍스트 영역 전체 길이와 모든 제어 코드 오프셋을 보존 → 0x08로 재압축 → (섹션 offset/length)
헤더를 갱신하며 컨테이너 재조립 → 컨테이너 엔트리를 파일 끝에 추가하고 레코드 갱신.
(`scen.rebuild_section_fit` 참고.)

## 6. 완성형(wansung) 한글 인코딩 (`culdcept/wansung.py`)

한글은 Shift-JIS에 없으므로, 필요한 한글 음절을 폰트에 이미 글리프 슬롯이 있는 **JIS
제1수준 한자 코드(0x889f~)** 에 하나씩 결정적으로 배정합니다. 그런 다음 그 슬롯에 한글
글리프를 그려 넣고(§4), 번역문을 그 SJIS 코드로 인코딩합니다. 게임은 평소처럼 코드→CMAP→
글리프 조회로 한글을 그려냅니다.

한계: 재활용한 한자 코드가 아직 번역하지 않은 화면에 나타나면 그 자리에 한글이 보입니다
(부분 패치의 알려진 특성). 한 장면을 완전히 번역하면 그 장면 안에서는 충돌이 없습니다.

## 7. 나레이션 텍스처 (양피지 가이드, `apply_narration.py`)

시작화면·튜토리얼·덱선택·배틀중단에서 나오는 양피지 가이드 나레이션은 폰트로
렌더하는 텍스트가 아니라 **3DS 네이티브 ETC1A4 텍스처**로, 시나리오 컨테이너(0x0d)
안에 헤더 없이 raw 픽셀로 저장돼 있다. 크기는 **256×64**(1~2줄)와 **256×128**(3~4줄).

- **스위즐**: PICA200 표준. 8×8 픽셀 타일이 row-major로 나열되고, 타일 안은 4×4 블록
  4개가 2×2 row-major. ETC1A4 한 블록 = 16바이트(**알파 A4 8바이트 + 색 ETC1 8바이트**).
  블록 `(bx,by)`의 알파 시작 오프셋:
  `off = TS + ((by//2)*32 + bx//2)*64 + ((bx%2)+(by%2)*2)*16` (256폭 기준, tiles-per-row=32).
- **알파(A4)**: 블록당 8바이트 = 16픽셀×4비트. 니블 순서는 column-major
  (`픽셀 i → x=i//4, y=i%4`, 저니블 먼저). 이 알파가 글자 모양, 색(ETC1)이 글자 색.
- **한글 주입**: 알파만 한글 글리프로 교체하고, 색 블록은 균일 진회색 ETC1
  (`33 33 33 00 00 00 00 00` ≈ #353535)로 채운다 → 원본과 같은 진회색 글자.
  ETC1 인코더가 필요 없다(색이 균일하므로).
- **겹침 보존**: 한 텍스처의 안 보이는 아래쪽에 다른 텍스처(예: 덱 속성 설명)가
  바이트를 공유하는 경우가 있다. **한글 잉크가 있는 행까지만** 덮고 그 아래는
  건드리지 않으면 두 텍스처가 공존한다.
- **정렬**: 게임은 화면마다 표시창 위치가 다르다(설정=풀, 배틀=중앙). 각 줄을 **원본
  일본어 잉크의 가로 중심**에 맞춰 렌더하면 잘림 없이 표시된다.
- **주입 방법**: 컨테이너를 디코드(0x0d는 게임 code.bin 기반 디코더 필요)→알파 교체→
  `0x08` 무압축으로 재인코딩(커스텀 LZMA 재압축 회피)→파일 끝에 append + 오프셋 갱신.

번역 데이터는 `narration_ko.json`(엔트리·오프셋·크기·한글 줄), 로직은
`apply_narration.py` 참고. 위치는 에뮬레이터 텍스처 덤프와 알파 블록 정확 일치로 특정한다.

## 8. 비압축(raw, type 0x00) 텍스트 섹션 주의

일부 시나리오 컨테이너 섹션은 압축(0x08) 대신 **raw(0x00)** 로 저장된다(예: 퀘스트2
1947.s2 — 제목·설명·노드명). 게임은 이를 **null(0x00) 분리 세그먼트 순서**로 읽는다
(`[제목][바로 다음=설명][노드…]`). 따라서 번역 시 세그먼트 개수·순서를 보존해야 한다.
번역문이 원문보다 짧다고 **제목을 null로 패딩하면 빈 세그먼트가 생겨** 게임이 그 다음
필드(설명)를 빈칸으로 읽는다. 길이 차이는 **맨 끝(포인터 테이블 앞)에서** 흡수하고,
제목→설명은 반드시 연속 세그먼트로 둔다.
